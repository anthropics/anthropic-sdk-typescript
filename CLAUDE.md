# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository overview

This is `@anthropic-ai/sdk` (anthropic-sdk-typescript), the official TypeScript SDK for the Claude API. Most of the SDK is generated by Stainless from the OpenAPI spec referenced in `.stats.yml`. The generator never modifies `src/lib/` or `examples/` — hand-written code belongs there. Manual edits elsewhere persist between generations but can produce merge conflicts with generator output.

## Commands

Requires yarn v1.

- Install: `yarn`
- Build: `yarn build` — builds the root SDK into `dist/`, then builds every package in `packages/*`
- Test: `yarn test` — auto-starts a Steady mock server on port 4010 against the OpenAPI spec, runs root jest, then each package's tests. Set `TEST_API_BASE_URL` to test against a different server.
- Single test file: `yarn test tests/streaming.test.ts` (args are forwarded to jest, but package suites still run afterwards). To run only one file, keep a mock server running (`./scripts/mock --daemon`) and invoke jest directly: `./node_modules/.bin/jest tests/streaming.test.ts -t "test name"`. Tests under `tests/api-resources/` require the mock server; pure unit tests do not.
- Lint: `yarn lint` — prettier check, eslint, a full build, tsc, arethetypeswrong, publint (slow because it builds)
- Auto-fix format + lint: `yarn fix`
- Run an example: `yarn tsn examples/<file>.ts`

Jest maps `@anthropic-ai/sdk` imports to `src/`, so tests exercise the TypeScript source directly — no build step needed to run tests.

## Architecture

- `src/client.ts` — the main `Anthropic` client. `src/resources/` — generated API resources and types (messages, models, completions, beta). The full generated API surface is indexed in `api.md`.
- `src/core/` — public, non-resource SDK infrastructure: streaming, pagination, errors, uploads, `APIPromise`, middleware, credentials.
- `src/internal/` — private modules; not importable by users and free to change between releases.
- `src/lib/` — hand-written code the generator never touches: `MessageStream`/`BetaMessageStream`, tool runners (`src/lib/tools/`), credential chain (`src/lib/credentials/`), sessions, environments. The streaming/tool-runner helper API is documented in `helpers.md`.
- `src/helpers/` — Zod and JSON-schema tool-definition helpers.
- `src/tools/` — agent-toolset and memory tools; each Node implementation (`node.ts`) has a `node.browser.ts` shim wired through the package.json `browser` field (see Node-only code below).
- `packages/` — separately published wrapper SDKs (`bedrock-sdk`, `vertex-sdk`, `foundry-sdk`, `aws-sdk`). They depend on the root package via `file:../../dist/`, so run `yarn build` at the root before working in a package.
- Publishing happens from `dist/`, never the repo root: the build copies `src/` into `dist/` and rewrites package.json so subpath imports like `@anthropic-ai/sdk/resources/foo` resolve.

## Node-only code

Bundlers follow every statically-resolvable import — including lazy `import('./relative/path')` — and fail browser builds on `node:` builtins in the resulting chunks.

- Code referencing Node builtins must live in a module named `node.ts` (or a `node/` directory). Everything else in `src/` must be runtime-agnostic.
- Strongly preferred: Node-only modules are imported by the user and passed into the SDK. SDK-internal code must not import them, statically or dynamically.
- If an internal reference is unavoidable, shim the module via the package.json `browser` field: a `<name>.browser.ts` with an identical export surface where every value export throws `<exportName> requires Node.js or a Node-compatible runtime`. Add runtime and type-level parity tests, and keep the internal import lazy.
- Never hide imports from bundlers (`webpackIgnore` comments, variable specifiers, `eval`). Never reference `process` at module scope in statically-reachable code without a `typeof process` (or `globalThis.process?.`) guard.
- `import type` of Node builtins is fine anywhere — types are erased.
